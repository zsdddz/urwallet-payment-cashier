# 🌐 全球支付系统：动态组件与 UI 渲染架构规范 (V1.0)

## 一、 核心设计理念

1. **配置即 UI (Metadata-Driven UI)：** 前端不写死任何渠道特定的业务表单。所有业务页面的标题、描述、输入框、按钮组和交互动作，均由后端的 `checkout_schema` 动态下发。
2. **极简与克制 (Minimalist & Abstract)：** 遵循极简主义美学。去除多余的边框和复杂的色彩叠加，大面积留白。视觉焦点永远集中在“核心信息展示”和“下一步操作”上。
3. **高容错与平滑降级 (Fallback-First)：** 面对不同国家参差不齐的网络环境和设备，必须提供平滑的交互降级（如：DeepLink 唤起失败时，平滑降级为二维码展示或应用商店引导）。

## 二、 基础 UI 与排版规范

- **页面布局 (Layout)：** 采用纵向流式布局（Stack Layout）。元素自上而下排列，核心输入框与按钮保持全宽排版，留白充足，避免视觉拥挤。

  **色彩系统 (Color System - 转化率导向)：**

  - **主色 (Primary Color)：信任与促单色。** 强烈建议使用能传递安全感和吸引力的明亮色系（如：安全蓝、活力绿、品牌紫），或直接应用后端动态下发的渠道专属 `theme_color`。**严禁在收银台主视觉使用大面积的暗黑系。**
  - **辅助色 (Semantic Colors)：** 根据 `notice` 类型定义。信息蓝（INFO）、成功绿（SUCCESS）、警告橙（WARNING）、危险红（ERROR）。颜色需保持清透，避免脏色。
  - **底色 (Background & Surface)：** **绝对的明亮与清爽。** 页面主背景强制使用纯白 `#FFFFFF`。若需区分内容区块，使用极其轻微的冷灰底色（如 `#F7F9FA` 或 `#F2F4F7`）。通过高对比度确保金额和账号等核心信息清晰易读。

  **圆角与阴影 (Radius & Shadow)：** * **圆角：** 全局统一使用适中的圆角（如 `8px` 或 `12px`），传递现代感与亲和力。

  - **阴影：** 仅在悬浮倒计时或底部固定的“支付按钮”区域使用极轻的弥散阴影（Soft Shadow），增加页面的空间感和按钮的“可点击感”。

## 三、 提示与辅助组件体系 (Feedback System)

任何提示信息必须通过 JSON 节点配置，前端根据配置按需渲染。

### 1. 全局步骤提示 (`notice`)

用于页面顶部的强提醒。对应 JSON 中的 `notice` 对象。

| Type 值   | 视觉表现                         | 适用场景                             |
| --------- | -------------------------------- | ------------------------------------ |
| `INFO`    | 淡蓝色背景，可选搭配 ℹ️ 图标      | 步骤说明、倒计时提醒、中性通知       |
| `WARNING` | 淡橙色/黄色背景，可选搭配 ⚠️ 图标 | 风险提示、防封号要求、实名要求       |
| `ERROR`   | 淡红色背景，可选搭配 ❌ 图标      | 严重警告（如防骗提示、不要备注信息） |
| `SUCCESS` | 淡绿色背景，可选搭配 ✅ 图标      | 完成某个中间节点的反馈               |

### 2. 字段级辅助文本 (`helper_text`)

- **视觉表现：** 位于对应表单输入框的**正下方**，使用小号字（如 12px）和浅灰色（如 `#999999`）。
- **场景：** 用于补充说明输入规则（如“无需输入国家代码”、“请填写真实的12位流水号”）。

### 3. 字段级悬浮说明 (`tooltip`)

- **视觉表现：** 位于对应字段 Label 的**右侧**，显示为一个灰色的 `(?)` 或 `(i)` 小图标。点击或 Hover 时弹出气泡框。
- **场景：** 解释专业名词或敏感字段（如解释“平台安全PIN码是什么”），不占用页面宝贵空间。

## 四、 核心组件字典 (`ui_type` Specification)

这是前端必须实现并注册到动态渲染引擎中的组件库清单。后端下发 `ui_type`，前端映射为具体的 Vue/React/Native 组件。

### 🔘 1. 输入类组件 (Input Components)

| `ui_type`         | 前端组件映射        | 交互要求与表单表现                                           | 数据回传类型    | 常见业务场景                      |
| ----------------- | ------------------- | ------------------------------------------------------------ | --------------- | --------------------------------- |
| **`TEXT`**        | 单行文本输入框      | 支持 `placeholder`，支持通过 `regex` 字段进行前端失焦校验。  | String          | 真实姓名、卡商账号、付款人手机号  |
| **`NUMBER`**      | 纯数字键盘输入框    | 唤起移动端**数字键盘**，过滤任何非数字字符。                 | String          | OTP 验证码、手机号、12位 UTR 单号 |
| **`PASSWORD`**    | 密码掩码输入框      | 输入内容默认显示为 `*` 或黑点，右侧提供 👁️（显示/隐藏）切换图标。 | String          | 平台安全 PIN 码、接口授权密钥     |
| **`RADIO_GROUP`** | 单选框组 (纵向排列) | 根据 `options` 数组渲染列表。若配置了 `is_dynamic: true`，代表选项由上一步异步返回生成。 | String (选中值) | 选择可用的 UPI 账号、选择开户行   |

### 🖼️ 2. 展示类组件 (Display Components)

| `ui_type`            | 前端组件映射        | 交互要求与表单表现                                           | 适用业务场景                           |
| -------------------- | ------------------- | ------------------------------------------------------------ | -------------------------------------- |
| **`QR_IMAGE`**       | 二维码展示容器      | 接收一个 URL 或 Base64，渲染为二维码。**必须支持**点击放大、长按保存到相册。 | 加密货币充值码、静态收款码展示         |
| **`TEXT_COPY`**      | 只读文本 + 复制按钮 | 大号加粗字体展示重要信息。右侧强制附加“复制”图标，点击后触发系统剪贴板并弹出“已复制”轻提示。 | 展示收款钱包账号、UPI ID、TRC20 地址   |
| **`PLATFORM_LOGOS`** | 小图标流式矩阵      | 将 `options` 中的 `{icon_url, label}` 渲染为一排（或网格）带有圆角底色的小图标，仅做视觉展示，不可点击。 | 收银台提示“支持 PhonePe / GPay 等付款” |

### 👆 3. 互动与动作组件 (Action Components)

| `ui_type`            | 前端组件映射       | 交互要求与表单表现                                           | 适用业务场景                             |
| -------------------- | ------------------ | ------------------------------------------------------------ | ---------------------------------------- |
| **`IMAGE_UPLOAD`**   | 图片上传/拍摄控件  | 渲染为一个虚线外框，中心有 ➕ 图标。点击唤起手机相册或相机。选中图片后，先调用云存储上传接口，获取真实的 OSS/S3 URL 赋给该字段。 | 绑卡时上传个人收款码、收银台提交付款截图 |
| **`DEEPLINK_GROUP`** | 快捷拉起应用按钮组 | 将 `options` 渲染为醒目的长条按钮。点击时提取 `value` (如 `phonepe://`)，使用系统级方法尝试唤醒本地 App，并附带倒计时降级逻辑。 | 收银台一键拉起本地支付 App               |

## 五、 状态机与动作调度 (`action_type` 规范)

每一个完整的 `step` 节点，除了包含 UI 元素外，都会包含一个 `action_type`。这是告诉前端在点击页面底部的“提交/下一步”按钮时，后端需要执行什么意图的代码分支。

*前端逻辑：* 点击提交时，统一向网关发起 `POST /api/bind/submit` 或 `POST /api/checkout/submit`。在 Request Body 中携带当前 `step` 的 `action_type` 和收集到的 `stepData` 键值对。

**标准定义的 Action 动作库：**

1. **`VERIFY_IDENTITY` (身份验证)：** 多步绑卡的第 1 步。触发后端调用 RPA 去发送短信或校验账号存活。
2. **`SUBMIT_OTP` (提交验证码)：** 多步绑卡的第 2 步。触发后端携带验证码去请求渠道 API 获取凭证。
3. **`SELECT_UPI` (选择账号)：** 多步绑卡的最后一步。确认选中项，完成账单关联，转正数据库状态。
4. **`SUBMIT_STATIC_BIND` (静态码绑定)：** 弱交互渠道的单步绑定。直接将文本和截图写入数据库。
5. **`SUBMIT_PAYER_INFO` (收银台-提交付款人信息)：** 中东/非洲模式收银台。先让用户输入自己手机号进行查账定位。
6. **`SUBMIT_UTR` (收银台-提交流水号)：** 印度模式收银台。提交12位交易回执单号。
7. **`SUBMIT_PROOF_IMAGE` (收银台-提交图片凭证)：** 提交转账成功截图，触发后续的 TG 人工审核流水。

🧱 一、 核心字典表 (Dictionary)
在记录具体 JSON 之前，先规范化我们体系中的三个核心枚举，这是前后端对齐的基石。

1. 视图容器布局 (ui_layout)
决定前端在解析该 step 时，使用什么样的外层容器来包裹表单元素。

STANDARD_PAGE: 全屏标准页面（默认态，包含顶部大额数字、倒计时）。

BOTTOM_DRAWER: 底部弹出的半屏抽屉（常用于拦截或前置鉴权，背后主视图会增加半透明遮罩）。

2. 交互动作指令 (action_type)
决定付款人点击“提交/下一步”时，请求后端 /app/checkout/submit 接口的底层路由分支。

SUBMIT_PAYER_INFO: 提交付款人身份信息（如手机号、姓名），用于反爬防抓和对账前置。

SUBMIT_UTR: 提交银行/第三方支付流水号。

SUBMIT_HASH: 提交加密货币区块链上的交易 TXID。

SUBMIT_PROOF_IMAGE: 提交转账成功截图（通常意味着该笔订单进入待审核终态）。

3. 元素渲染类型 (ui_type)
TEXT_COPY: 带有复制按钮的文本（用于展示收款账号、银行代码）。

QR_IMAGE: 占位/静态二维码图形。

DEEPLINK_GROUP: 一键拉起本地 App 的图标组。

PLATFORM_LOGOS: 纯视觉展示的支持平台矩阵。

INPUT: 文本/数字输入框。

IMAGE_UPLOAD: 图片上传控件。

缓存判定：
前端在渲染前，先检查浏览器的 localStorage.getItem("user_payer_phone") 是否有值。

如果有值：说明用户之前输入过，直接忽略 interceptor，正常渲染主页面。

如果无值：正常渲染主页面，但在主页面上方盖一层 高斯模糊遮罩 (Backdrop Blur)，并居中弹出 interceptor 里配置的表单。

拦截器提交：
用户在弹窗中输入手机号并点击“确认”时，前端调用 /app/checkout/submit (携带 action_type: ASYNC_SUBMIT_PAYER) 将手机号发给后端。
成功后，前端执行 localStorage.setItem("user_payer_phone", "用户输入的号码")，随后关闭弹窗，撤销模糊遮罩，让用户继续传截图。

最终提交：
用户传完截图点击最下方的提交时，前端再次调用接口 (携带 action_type: SUBMIT_PROOF_IMAGE)。你可以让前端顺便把刚才缓存的手机号一起带上，确保万无一失。

# 多语言实现补充补充

在这个“后端驱动渲染（Server-Driven UI）”的架构中，实现多语言有一个核心准则：后端负责翻译动态配置，前端负责翻译静态骨架。 绝对不要让前端去翻译后端下发的 JSON，否则前端逻辑会极其沉重。

下面为你梳理一套标准的多语言实现方案，特别是考虑到你要做中东（埃及/伊拉克）市场，我会在最后补充一个极其关键的防坑指南。

一、 语言上下文的识别与下发 (Context Detection)
收银台是给匿名 C 端用户使用的，系统需要智能判断该显示什么语言，优先级顺序如下：

强行指定 (最高优先级)：商户下单时，在 URL 上拼装 ?lang=ar 或 ?lang=en。

浏览器默认语言：读取 HTTP 请求头中的 Accept-Language。

渠道国家兜底 (最低优先级)：如果前面都没获取到，根据该订单所属的通道国家（如 EG_VODAFONE 属于埃及），自动回退到默认语言（阿拉伯语 ar）。

二、 后端动态配置的多语言处理 (BFF 层替换)
在你的数据库 app_payment_channel 中，checkout_schema 不应该存死某种语言的明文，而是使用双大括号占位符 {{...}} 作为多语言 Key。

1. 数据库存储的 JSON 示例 (运营配置侧)
JSON
[
  {
    "step": 1,
    "title": "{{i18n.checkout.title.upload_proof}}",
    "description": "{{i18n.checkout.desc.transfer_to_wallet}}",
    "elements": [
      {
        "field_name": "payer_phone",
        "label": "{{i18n.checkout.label.payer_phone}}",
        "ui_type": "NUMBER",
        "placeholder": "{{i18n.checkout.placeholder.enter_phone}}"
      }
    ]
  }
]
2. 后端 BFF 层的替换逻辑
当 /app/checkout/info 接口接收到请求时，代码需要做以下处理：

确定用户的目标语言（如 ar）。

从数据库取出 checkout_schema 的 JSON 字符串。

调用系统的多语言组件（如 Spring Boot 的 MessageSource 或 Redis 中的多语言字典）。

正则正则替换：扫描 JSON 字符串中的所有 {{i18n.xxx}}，替换成目标语言的真实文本。

将替换后的纯文本 JSON 下发给前端。

这样，前端拿到的就已经是阿拉伯语或英语的最终结果，前端完全无感知，直接“无脑渲染”。

三、 前端静态骨架的多语言处理
前端 (Vue/React) 内部只需引入 vue-i18n 或 react-i18next，用来翻译那些不被后端 JSON 控制的死静态元素。

例如：

倒计时组件的单位：14 Minutes 59 Seconds -> 14 分 59 秒

全局错误提示：Network Error -> 网络异常，请重试

接口请求的 Loading 状态：Loading... -> جار التحميل...

除了这些，收银台中心区域的所有文字，前端都直接展示后端下发的字符串即可。

⚠️ 四、 中东/非洲市场的致命暗坑：RTL (Right-to-Left) 支持
既然你明确提到了伊拉克和埃及，就绝对绕不开阿拉伯语。阿拉伯语不仅是文字不同，它的书写和阅读方向是从右向左 (Right-to-Left, RTL) 的。

如果在阿拉伯语下，你的页面还是左对齐，当地人看起来会像是在看镜子里的倒影，完全无法使用。

架构级解决方案：

后端在 /info 接口返回数据时，除了返回替换好的 JSON，必须额外返回一个页面方向的标识 (Direction)：

JSON
{
  "code": 200,
  "data": {
    "order_info": { "..." },
    "ui_config": {
      "language": "ar",
      "direction": "rtl" // 英文/印地语返回 "ltr" (Left-to-Right)
    },
    "checkout_schema": [ "..." ] // 这里已经是全阿拉伯语的JSON了
  }
}
前端的配合动作：
前端拿到这个 direction: "rtl" 后，只需要做一件事：给整个收银台的根部 HTML 标签挂上这个属性。

HTML
<div dir="rtl" class="checkout-container">
   </div>
现在的主流前端框架组件（如 Element/Ant Design）和 CSS Flexbox 只要检测到父级有 dir="rtl"，会自动把输入框光标移到右侧，把所有的左右布局镜像反转，无需前端写两套 CSS。
